# Skill: GitHub Issue Manager

GitHub Issueのライフサイクル全体（作成、開発開始、実装、レビュー、完了）を管理し、Issue駆動開発（IDD）のプロセスを自動化・支援します。

## Capabilities

1.  **Create Issue**: ユーザーの要望に基づき、適切なテンプレートとラベルを使用して新規Issueを作成します。
2.  **Manage Development Cycle**: 既存のIssueに対し、仕様策定(Start) -> 実装(Task) -> レビュー(Review) -> 完了(End) の一連のフローを実行します。

---

## Usage Guidelines

このスキルは開発プロセスを定義しますが、具体的な技術スタック（テストコマンド、リンター、ディレクトリ構成、ブランチ命名規則等）については、必ず **`AGENTS.md`** を参照して、プロジェクト固有のルールに従ってください。

### 1. Create Issue (新規Issue作成)

ユーザーから要望があった際、プロジェクトのテンプレート（`.github/ISSUE_TEMPLATE/` 等）に基づき、分類、下書き作成、ユーザー確認を経てIssueを作成します。

---

### 2. Manage Development Cycle (IDDワークフロー)

#### A. 開発開始 (Phase: Start)

1.  **環境準備**:
    *   `AGENTS.md` の「3.1. Issue-Driven Development (IDD)」等の記述に従い、作業ブランチの作成とIssue情報の取得を行ってください。
    *   プロジェクト固有のセットアップスクリプトがある場合は、それを利用してください。

2.  **仕様ドキュメント生成**:
    *   取得した情報を基に、プロジェクト指定のディレクトリ（例: `docs/issues/[ID]/`）に仕様書を作成します。
    *   通常、`requirements.md` (要件), `design.md` (設計), `tasks.md` (タスク) の3点を含めます。

#### B. タスク実行 (Phase: Task)

1.  **実装ループ**:
    *   タスクリストを読み込み、順次実装します。
    *   **検証**: 各ステップで `AGENTS.md` 指定のテストコマンドを実行し、品質を確保してください。

#### C. 品質レビュー (Phase: Review)

1.  **自己レビュー**:
    *   変更内容が要件を満たし、`AGENTS.md` のコーディング規約や設計方針に合致しているか厳格に確認します。
    *   **アーキテクチャ監査**:
        - **レイヤー分離**: 共通基盤（`.gemini`, `.roo`）がプロジェクト固有領域（`.ops`）に依存していないか。
        - **配置ルール**: 定義とスクリプトが適切な `skills/` フォルダに集約されているか。`docs/` 等にロジックを置いていないか。
        - **ポータビリティ**: そのフォルダを別プロジェクトにコピーしても動作するか。

#### D. 開発終了 (Phase: End)

1.  **最終検証**:
    *   プロジェクト全体のテスト・ビルドコマンドを実行し、CIが通る状態であることを確認します。

2.  **記録と完了**:
    *   `AGENTS.md` の規定に従い開発ログを記録し、コミット・PR作成・マージ（または依頼）を行います。
